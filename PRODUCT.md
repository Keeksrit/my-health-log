# Product

## Register

product

## Platform

web

## Users

The primary user is a person tracking their own health day to day — logging symptoms (with a body map and severity), food and nutrition, medications, and lab results from a phone, throughout the day. The record is also read by a second audience: a clinician who reviews the data (for example, brought to an appointment) and must be able to make sense of it without a walkthrough. The app is being built so it can eventually serve other self-trackers beyond its original owner, so decisions should hold up for more than one person.

## Product Purpose

Health Log is a single personal health record that unifies what usually lives in five disconnected apps — symptoms, body map, food/nutrition, medications, and labs — into one timeline. It exists so that patterns become visible: the point isn't just to write things down, it's to be able to look back and see how food, medication, sleep, symptoms, and lab values move together over time. Success is when that data reveals correlations the user couldn't otherwise see, and when the accumulated history is trustworthy enough to reason over.

## Positioning

Every entry is structured data you can actually analyze — correlations, lab trends, nutrition breakdowns — not free-text journal entries that can only be re-read.

## Brand Personality

Calm but rigorous. The surface should feel warm, steady, and reassuring to return to every day; underneath, it takes data accuracy seriously and never dresses up guesses as facts. The voice is plain and precise — no hype, no gamified cheer, no wellness-influencer tone. Three words: calm, rigorous, humane. The feeling to leave the user with is quiet confidence that they are on top of their own health.

## Anti-references

It must never feel boring, generic, or forgettable — a health tracker only works if the person actually wants to open it, so dull-but-functional is a failure, not a safe default. It should also not read as a cold, sterile clinical EMR; the warmth is deliberate and distinguishes it from hospital software.

## Design Principles

Analyzable by design — capture must be fast, but everything captured is structured so it can be queried, charted, and correlated later. Speed at entry never costs structure at analysis.

Calm surface, rigorous core — the interface is reassuring and low-stress to use daily, while being visibly serious and honest about the data it holds.

Encode meaning redundantly — because the app leans on color (severity dots, food-type colors, body-map fills), meaning is always carried by more than color alone (shape, label, position), so it survives color blindness and a printed page.

Legible to a second reader — a clinician should be able to open the record cold and understand it. Clarity for someone who wasn't there when it was logged is a first-class concern.

Never boring — low friction plus a measure of genuine craft is what keeps the daily habit alive; the tool earns the return visit.

## Accessibility & Inclusion

Target WCAG AA contrast throughout (body text ≥4.5:1, large text ≥3:1), including placeholder and muted text. Because the interface is heavily color-coded, it must be color-blind safe: never rely on color as the only signal — pair it with shape, icon, label, or position. Honor reduced-motion preferences for any animation.
