# Nutrition Feature — Design

**Date:** 2026-06-25
**Status:** Approved (design), pending implementation plan

## Summary

Add a **Nutrition** page to the health-log app. It lets the user build a personal
**food library** and log what they eat against it. A food is defined by its exact set
of ingredients: the same name maps to a fixed ingredient set, and a different set is a
different food (e.g. *Pepperoni Pasta* = pepperoni + coconut milk + pasta vs
*Pepperoni Pasta 2* = pepperoni + heavy cream + pasta).

Data lives in the existing Supabase `health` schema. The feature mirrors the proven
structure of the medications feature (library tables + log table, referenced by foreign
key, with a dedicated `lib/*.ts`, `types/*.ts`, and modal-driven pages).

## Data model (`health` schema)

```sql
nutrition_ingredients (
  id          uuid primary key default gen_random_uuid(),
  name        text not null unique,          -- matched case-insensitively
  type        text,                           -- nullable (optional / stubs)
  created_at  timestamptz not null default now()
)
-- type ∈ {Grains & Starches, Proteins, Dairy, Dairy alternative,
--         Fruit, Vegetable, Fats & Oils, Processed} | null

nutrition_foods (
  id          uuid primary key default gen_random_uuid(),
  name        text not null unique,          -- matched case-insensitively
  type        text,                           -- nullable (meal category, optional)
  created_at  timestamptz not null default now()
)
-- type ∈ {Breakfast, Lunch, Dinner, Snack, Drink} | null

nutrition_food_ingredients (                  -- link table (many-to-many)
  food_id        uuid not null references nutrition_foods(id) on delete cascade,
  ingredient_id  uuid not null references nutrition_ingredients(id) on delete restrict,
  primary key (food_id, ingredient_id)
)

nutrition_consumption_log (
  id          uuid primary key default gen_random_uuid(),
  food_id     uuid not null references nutrition_foods(id) on delete restrict,
  amount      numeric not null,
  unit        text not null,                  -- ∈ {g, ml, serving, piece}
  eaten_at    timestamptz not null,
  created_at  timestamptz not null default now()
)
```

### Model rules

- **`id` is a UUID** primary key (the "Index"), generated by the database — not a
  user-typed sequential number. Matches the medications tables.
- **`name` is unique and matched case-insensitively** on both foods and ingredients.
  This is what keeps set-matching reliable so "coconut milk" and "Coconut Milk" are not
  treated as different ingredients. Enforce with a unique index on `lower(name)`.
- **The consumption log references a food by `food_id` (foreign key)**, never by name.
  The food's name/type are read via the relation. Renaming a food never orphans past logs.
- **Only a name is mandatory** when creating a food: `type` and ingredients are optional.
- **Ingredient `type` is optional** when creating an ingredient.
- `nutrition_` prefix on all four tables so they group cleanly and can be joined later.

## Page structure

One `/nutrition` route with two tabs; all add/import actions are modals (matching the
existing meds/symptom flows). One bottom-nav entry.

```
Nutrition
[ Log ] [ Library ]

Log tab:      + Log entry        list of recent consumption entries (edit / delete)
Library tab:  + Food  + Ingredient  ⬆ Import CSV
              list of foods (open → detail)  ·  list of ingredients
```

## Data flow — manual actions

### Log an entry (Log tab — the daily action)
Pick a food from a searchable list (by name) → enter `amount` + `unit` → pick `eaten_at`
(defaults to now) → insert into `nutrition_consumption_log`. Recent entries are listed
below with edit/delete.

### Add a food (Library tab)
1. Enter ingredients via **autocomplete from `nutrition_ingredients`** (chips).
   New ingredients can be created inline (name; type optional).
2. As the ingredient set changes, check existing foods for an **exact set match**.
   If one exists, surface it ("this is *Pepperoni Pasta*") and offer to open it instead
   of creating a duplicate.
3. If no match, the user provides a **name** (required) and optional meal `type`, then
   saves — writes a `nutrition_foods` row plus `nutrition_food_ingredients` links.
   A food may be saved with no ingredients (only name is mandatory).

### Food detail
Shows the ingredient list **and the distinct set of their types**, e.g.
*Pepperoni Pasta → pasta, pepperoni, cream · Grains & Starches, Proteins, Fats & Oils*.

### Set-matching
Runs client-side: fetch foods with their ingredient-id arrays (via the link table join)
and compare sets in JS. Adequate for a personal-scale dataset; no special SQL required.

## CSV import (three formats, lenient)

A modal where the user picks the format, then uploads a `.csv` file or pastes text.

| Format      | Columns                          |
|-------------|----------------------------------|
| Ingredients | `name, type`                     |
| Foods       | `name, type, ingredients` (ingredient names comma-separated inside a quoted cell) |
| Log         | `food, amount, unit, eaten_at`   |

Flow: parse → **preview table** → confirm → insert.

**Lenient auto-create:** rows that reference an unknown ingredient or food create that
record as a **stub** (type/ingredients left unset) and link it. Nothing is skipped for a
missing reference. An **import summary** reports counts and lists every stub created so
the user can complete them later. Rows that are genuinely unparseable (bad structure,
non-numeric amount) are reported in the summary, not silently dropped.

**Parser:** a small dependency-free CSV parser handling double-quoted fields (needed for
the ingredients cell). No `papaparse` dependency added.

## Error handling

- Supabase errors surface as visible messages inside each modal (improving on the
  current console-only pattern).
- Duplicate-name / unique-constraint conflicts are caught and explained
  ("an ingredient named X already exists").
- `amount` must be a positive number; `unit` and `type` are constrained to their lists.

## Testing

The project currently has no test suite. Add **light unit tests only for the three pure
helpers** (`parseCsv`, `matchFoodByIngredientSet`, `distinctIngredientTypes`) since they
are logic-heavy and error-prone. UI is verified manually, consistent with the rest of the
app.

## Out of scope (YAGNI)

- Nutrition figures / calories / macros.
- Stats-page charts for nutrition.

Both are straightforward to add later on top of `nutrition_consumption_log`.
